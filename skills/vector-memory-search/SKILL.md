# Vector Memory Search

Use this skill when you need semantic similarity search over agent memories — comparing new findings to past ones, building context from prior rounds, or searching for related patterns.

## When to Use

- Triaging orbit loop findings against historical patterns
- Searching for similar bugs across prior rounds
- Building context for agents from stored memories
- Deduplicating findings by semantic similarity

## Architecture

The hybrid backend combines two storage layers:

- **SQLite** — structured queries by agent, type, time range, metadata
- **AgentDB** — HNSW-indexed vector search for semantic similarity

Embeddings are generated by the EmbeddingProvider chain:

1. RuVector (if `@ruvector/embeddings` is installed)
2. Built-in deterministic hash embeddings (always available)

## Usage Patterns

### Store a finding with embedding

Store content + embedding so it is searchable by both SQL filters and vector similarity.

### Query by metadata

Use SQL-backed queries for structured lookups: by agent, type, time range, domain.

### Vector similarity search

Find the k most similar memories to a query embedding. Useful for:

- Comparing a new bug report against all prior bug reports
- Finding related fixes for a known issue pattern
- Detecting regression candidates from fixed issues

### Hybrid search

Combine SQL filters (e.g., domain = 'auth') with vector similarity to narrow results.

## Rules

1. **ALWAYS store embeddings** alongside memories that will need semantic search later.
2. **ALWAYS use hybrid search** when you know the domain — it is faster and more precise than pure vector search.
3. **NEVER rely solely on vector search for exact-match lookups** — use SQL queries for those.
4. **ALWAYS check the embedding provider** — if only the built-in provider is active, similarity is based on character-level hashing, not true semantics. This is sufficient for grouping but not for nuanced meaning comparison.
